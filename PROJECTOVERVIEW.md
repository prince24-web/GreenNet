# GreenNet — Technical Project Overview

This document provides a detailed breakdown of GreenNet's technical architecture, component design, offline data flows, and performance optimizations. It is designed to help hackathon judges and developers understand how the system achieves reliable, high-speed, local-only AI inference on resource-constrained target devices.

---

## 🏗️ System Architecture & Data Flow

GreenNet uses a modular, decoupled architecture consisting of a lightweight web-view frontend and a Python-powered orchestrator coordinating relational databases, vector storages, and ONNX runtime inference on the CPU.

## 🧩 Deep Dive Into Core Components

### 1. The Frontend UI Layer (`ui/`)
GreenNet's user interface is designed to look premium, modern, and earthy, while staying highly functional:
* **UI Shell (`Index.HTML`)**: Uses responsive CSS grids. Includes a sidebar showing model status details, plant image attachment previews, quick-action query chips, and a scrolling chat log.
* **Earthy Theme (`Style.CSS`)**: Avoids standard flat browser designs. Employs a forest-green color palette, rounded glassmorphism cards, micro-animations, and custom skeleton loading states with pulsing indicator rings.
* **JS Bridge Controller (`App.JS`)**:
  * Listens for the `pywebviewready` event to initialize secure bindings.
  * Dispatches async tasks to the Python background worker thread via `window.pywebview.api.handle_query_stream()`.
  * Monitors custom window-level events (`agrisense:stream_start`, `agrisense:stream_token`, `agrisense:stream_end`, `agrisense:stream_error`) generated by Python.
  * Incorporates a lightweight, secure offline **markdown renderer** to print bullet points, tables, bold text, and lists correctly in chat bubbles.

### 2. Desktop Wrapper & Streaming API (`app.py`)
Instead of packaging the application inside a resource-heavy Electron shell (which consumes up to 800MB–1.5GB of RAM), GreenNet wraps the UI in **`pywebview`**:
* Spawns a native OS window container utilizing the system's underlying web engine (WebKitGTK on Linux/Ubuntu, WebView2 on Windows), reducing memory overhead to only ~80–150MB.
* Dispatches user queries to a **background daemon thread** to prevent locking or stuttering the main UI render loop.
* Utilizes `requests.post()` with `stream=True` to fetch tokens dynamically from local Ollama endpoints.
* Evaluates dynamic JavaScript in the browser window (`window.evaluate_js(...)`) to push incoming tokens to the DOM in real-time.

### 3. Central Context Coordinator (`src/orchestrator.py`)
This is the central routing center of the app. It takes user text and optional base64 image data, queries individual sub-systems, and consolidates findings into a single structured context block:
* **Image Processor**: If an image is present, it calls the vision sub-system. If a specific crop disease is diagnosed, it queries the SQLite database for active treatments for that disease.
* **Text Parser**: Scans the input text for mentions of crops or markets. If a crop or region is detected, it does a relational SQLite look-up for current Naira market pricing.
* **Context Compiler**: Joins RAG text blocks, SQLite listings, and vision diagnostics, prepends the expert system prompt, and builds the final consolidated instructions sent to the local LLM.

### 4. Vector Search & RAG (`src/rag.py` & `src/llm.py`)
To prevent the model from hallucinating or proposing non-Nigerian farming instructions, the orchestrator retrieves matching data blocks from a local vector store:
* Uses **ChromaDB** running in persistent disk-bound storage mode.
* Documents are vector-embedded using the `nomic-embed-text` model served locally by Ollama.
* Vector queries use **Cosine Similarity** to match the semantic meaning of the farmer's query, extracting the top 2 matching documents with a threshold distance constraint (`distance < 0.7`).

### 5. Relational SQLite Database (`src/database.py`)
For strict, tabular information (such as crop spacing, planting/harvesting calendar months, fertilizer ratios, NPK weights, regional market prices, and chemical pesticide brand names), vector search is unreliable.
* Uses Python's native **`sqlite3`** module, which loads rapidly and has no background database service overhead.
* Incorporates search helper methods using SQL `LIKE` queries matching crop names, local names (Hausa, Igbo, Yoruba, Pidgin), and symptom keywords.
* Automatically links pest symptoms to appropriate local treatment recipes and stores pricing estimates in Nigerian Naira (₦).

### 6. ONNX Crop Vision Diagnostics (`src/vision.py`)
GreenNet allows farmers to analyze crop leaf photographs locally:
* Uses **`onnxruntime`** to load a quantized MobileNetV3 neural network model (`~20MB` in file size) running entirely on CPU.
* Images are loaded via `Pillow`, normalized to ImageNet mean/standard deviations, resized to `(224, 224, 3)`, and converted to a float32 NCHW tensor.
* If `onnxruntime` is not set up, or the model file is not downloaded, the module falls back to a **rule-based color pattern analyzer**. It extracts mean channel values to detect yellowing patterns (typical of mosaic virus or nitrogen deficiency), dark brown spots (blight indicators), or pale gray patches (powdery mildew indicators), ensuring the user always gets a response.

## ⚡ Offline Performance Optimizations

Running LLMs on standard consumer hardware requires aggressive tuning. GreenNet implements the following design configurations:

1. **Model Pre-Heating (Startup Warmup)**:
   Ollama normally loads models on-demand, causing a 30–60 second cold-start latency on the first message. App startup triggers a background daemon thread that calls Ollama's API with a tiny dummy prompt and `keep_alive: -1`. This ensures both the LLM and Embedding models are fully loaded into system RAM *before* the user sends their first message.
2. **Conversational Short-Circuit**:
   Generating embeddings for short social inputs (such as *"Hello"*, *"Good morning"*, or *"Thank you"*) is a waste of processing power. GreenNet scans queries under 40 characters for agricultural keywords. If none are found, it skips vector retrieval completely, reducing reply times for casual greetings to under 2 seconds.
3. **Context Reduction**:
   We restrict the LLM context size (`num_ctx`) to `1024` tokens and maximum generation length (`num_predict`) to `300` tokens. This yields faster processing times on CPU cores without sacrificing advice depth.
4. **Indefinite Keep-Alive (`keep_alive: -1`)**:
   Standard Ollama setups unload models from memory after 5 minutes of inactivity. We set the parameter to `-1` to retain models in RAM throughout the desktop session, ensuring consistent sub-3-second responses.
